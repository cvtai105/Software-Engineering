oauth là tiêu chuẩn thiết kế cho phép ứng dụng này truy cập tài nguyên của cùng 1 người dùng ở ứng dụng khác

Luồng thực thi:
VD1: truy cập vào ggdriver
    Client yều cầu người dùng cấp phép dùng driver
    người dùng accept thông qua Google Auth server
    Google Auth gửi authorization code thông qua callback url
    Client sử dụng authorization code gửi tới Google Auth server request accesstoken
    Client gọi api tới ggdriver kèm access token

    Câu hỏi: tại sao phải chia ra 2 loại authorization code và accesstoken, mà không gửi accesstoken ngay từ đầu

VD2 (phổ biến hơn): ứng dụng (App1) truy cập thông tin người dùng từ bên thứ 3 (App2)
    App1 cần đăng ký trong App2 với định danh, domain, callback url
    App2 kiểm tra và trả client_id, client_screte

    User thực hiện thêm dữ liệu app2 sang app1.
    App1 redirect sang app2 với message chưa client_id, redirect_URI, response_type, scope.
    App2 trả authorization code
    App1 gửi authorization code , client_id, secret_id, grant_type tới authorization server của app2
    app2 trả accesstoken
    app1 dùng accesstoken để fetch resource

OpenID connect: dùng trong trường hợp cần xác thực định danh từ bên thứ 3 của end-user và thông tin profile (không cần phải tự đăng ký thông tin thủ công)
Luồng thực thi:
    End user chọn sign-in bằng bên thứ 3, ứng dụng redirect tới OpenID Provider
    OpenId Provider tiến hành xác thực
    OpenID Provider trả code dùng 1 lần cho browser qua redirect_URI
    Server gửi code này tới OpenID Provider và lấy thông tin
